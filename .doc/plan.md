# PRD 기반 작업(Task) 구성

참고 문서: [.doc/PRD.md](.doc/PRD.md)

## 제품 범위 요약

- **목적**: 건축구조안전 모니터링 업무 통합(대장·자료 분배·검토서·단계 추적).
- **역할(가변)**: 모니터링 팀장 1, 총괄간사 1, 간사 5, 검토위원 50.
- **핵심 객체**: 관리번호(`2026-xxxx`), 통합관리대장(엑셀), 설계도서 폴더, 검토서(단일 시트 엑셀), 단계(예비검토 → 1~N차 보완).

```mermaid
flowchart LR
  subgraph phase0 [Phase0]
    MgmtNo[관리번호_부여]
    Ledger[통합관리대장]

  end
  subgraph phase1 [Phase1]
    Intake[접수폴더]
    Classify[자료분류_폴더복사]
    Notify[카카오_알림]
    ReviewerUI[검토대상_확인]
  end
  subgraph phase2 [Phase2]
    Upload[검토서_업로드]
    Validate[유효성검증]
    Store[클라우드_저장]
    Extract[대장_자동기입]
  end
  MgmtNo --> Ledger
  Ledger --> ReviewerAssign
  ReviewerAssign --> Intake
  Intake --> Classify
  Classify --> Notify
  Notify --> ReviewerUI
  ReviewerUI --> Upload
  Upload --> Validate
  Validate --> Store
  Store --> Extract
```

## Epic A — 기반 설계 (스택·보안·배포)

PRD에 스택이 없으므로 먼저 결정이 필요합니다.

- **A1**: 웹/데스크톱 여부(총괄간사 PC 로컬 폴더 `D:/...` 연동 시 데스크톱 또는 에이전트 필요).
- **A2**: 인증·역할(RBAC): 팀장 / 총괄간사 / 간사 / 검토위원 권한 매트릭스 초안.
- **A3**: 감사 로그(누가 언제 어떤 관리번호·단계에 무엇을 했는지).
- **A4**: 배포 대상(OS, 내부망 여부) 및 비밀 관리(카카오 API 키, 클라우드 자격증명).

---

## Epic B — 데이터 모델 & “통합관리대장” 연동

PRD 표(접속 화면)와 Phase 0~2가 모두 대장과 연결됩니다.

- **B1**: 도메인 모델 정의: `관리번호`, `주소`, `연면적`, `층수`, `고위험군`, 단계별 컬럼(설계도서 배포, 검토서 제출, 미제출 사유, 검토 결과, 보완도서 배포 … N차까지 가변).
- **B2**: 통합관리대장 엑셀 스키마 확정(시트명, 열 이름, 관리번호 키, 검토위원 열).
- **B3**: 대장 읽기/쓰기 전략(직접 xlsx 조작 vs 중간 DB 동기화). 동시 편집·파일 잠금 리스크 정리.
- **B4**: “저장 폴더” 경로를 대장 또는 DB에 보관하는 규칙.

---

## Epic C — Phase 0: 관리번호 & 검토위원 배치

- **C1**: 관리번호 형식 검증(`2026-xxxx` 등 연도·자릿수 규칙).
- **C2**: 관리번호별 통합관리대장 행 생성/조회 UI(또는 임포트).
- **C3**: 총괄간사용 검토위원 지정 UI → 대장의 검토위원 열 반영(수기 프로세스를 소프트웨어로 옮길지 범위 확정).

---

## Epic D — Phase 1: 설계도서 접수·자료분류·알림

- **D1**: 접수 경로 규칙 반영: 기본 예시 `.../01.접수자료/.../YYYYMMDD/` (PRD 경로와 표기 불일치 `1. 예비검토` vs `예비검토` — 구현 전 하나로 통일).
- **D2**: **[자료분류]**: 원본 폴더 선택 + 저장(대상) 폴더 선택 → 관리번호 파싱(폴더명 prefix) → 검토위원별 하위 폴더 생성 → 파일 이동/복사.
- **D3**: 단계(예비검토 / 1차 보완 …)와 폴더 템플릿 매핑.
- **D4**: 카카오톡 API 연동: 검토위원별 “검토대상 추가” 알림(수신자 매핑, 실패 재시도, 발송 이력).
- **D5**: 검토위원 로그인 후 **[검토대상]** 목록(관리번호, 단계, 일정 메타데이터). 설계도서는 웹하드에서만 받도록 안내(본 시스템에 파일 저장 금지 정책을 UI/정책 문서로 명시).

---

## Epic E — Phase 2: 검토서 업로드·검증·저장·대장 반영

- **E1**: 검토서 업로드 API/UI(역할: 검토위원).
- **E2**: 유효성 검증 모듈:
  - 파일명 = 관리번호만(`2026-0001.xlsm` 등 확장자 규칙 포함).
  - 파일 내부 관리번호 필드와 파일명 일치.
  - 제출자(로그인 사용자)와 엑셀 “검토자 이름” 일치.
  - 실패 시 구체적 오류 메시지 후 재업로드.
- **E3**: 검증 통과 파일을 날짜 폴더 구조로 **Google Drive 또는 AWS S3** 중 선택한 저장소에 업로드(경로 규칙·권한·버킷/드라이브 설계).
- **E4**: 엑셀에서 검토 결과 등 필요 필드 **자동 추출** → 통합관리대장 해당 열·단계에 기입(매핑 테이블 정의).
- **E5**: “검토서 미제출 사유” 등은 미제출 워크플로가 PRD에 없음 — 필요 시 별도 입력 화면 여부를 기획 확정.

---

## Epic F — 반복 패턴(예비검토 → N차 보완)

- **F1**: 단계 상태머신: `예비검토` → `보완(1차)` → … , 각 단계별 동일한 컬럼 세트(배포/제출/사유/결과) 적용.
- **F2**: 메인 그리드(접속 화면 표)는 단계·차수에 따라 컬럼 동적 표시 또는 스크롤/탭 UX 설계.
- **F3**: 이전 단계 완료 조건과 다음 단계 오픈 규칙(업무 규칙으로 문서화).

---

## Epic G — 품질·운영

- **G1**: 총괄간사 PC에서 로컬 디스크 접근이 필요한 기능에 대한 설치형 클라이언트 또는 파일 워처 명세.
- **G2**: 통합 테스트: 샘플 폴더·샘플 검토서로 E2E 시나리오.
- **G3**: 권한 테스트(검토위원은 타인 대상 수정 불가 등).

---

## 우선순위 제안(의존성)

1. **A + B** (역할·대장 스키마·연동 방식) 없이는 UI/자동화가 흔들립니다.
2. **C → D2 → D5** (관리번호·분류·검토대상 확인 최소 루프).
3. **E2 → E3 → E4** (검증·저장·대장 반영).
4. **D4**, **F** (알림·다차수)는 코어 루프 이후 확장 가능.

---

## PRD에서 구현 전에 정리할 불명확 사항(태스크에 “결정” 항목으로 포함 권장)

- 통합관리대장의 **정확한 열 목록**와 검토서 시트의 **필드 이름**.
- **Google Drive vs AWS** 최종 선택(또는 단계적).
- 카카오 알림: **비즈메시지/알림톡** 등 채널·템플릿·수신 동의.
- 검토위원 **50명 스케일**에서 엑셀 단일 파일 병목 시 **DB 마스터 + 엑셀 export** 여부.

이 구성은 별도 이슈 트래커(Jira, GitHub Issues 등)에 Epic/Story로 옮기기만 하면 됩니다.
