# 카카오 API 권한 검수 사용 목적서 (초안)

> 카카오 디벨로퍼스 "카카오톡 친구 API" + "카카오톡 메시지 API" 권한 신청용 문서

## 📋 폼 입력용 — 신청 사유 및 활용 시나리오

> 카카오 디벨로퍼스 콘솔의 "신청 사유 및 활용 시나리오" 단일 텍스트 필드에 그대로 붙여넣기.

```text
[서비스 소개]
건축구조안전 모니터링 통합 시스템은 대한민국 건축법상 필수 절차인
건축구조 안전성 검토 업무를 수행하는 내부 협업 도구입니다.
사용자는 모니터링 팀장·총괄간사·간사(총 7명)와 검토위원 약 50명,
총 60명의 업무 관계자로 구성됩니다.


[기능 활용 목적]
1. 친구 API(friends): 간사가 검토위원에게 카카오톡 업무 알림을 발송하기
위해, 본 시스템에 로그인한 검토위원의 카카오 UUID를 획득하여 내부
매칭 테이블에 저장합니다. 매칭은 간사가 수동으로 1회 진행합니다.

2. 메시지 API(talk_message): 검토 대상 배정, 설계도서 접수, 보완 요청,
마감 임박 등 업무 상태 변화를 검토위원에게 카카오톡으로 전달합니다.
이메일·문자 대비 응답률이 높은 카카오톡이 업무 연락에 필수적입니다.

모든 메시지는 간사가 화면에서 버튼을 직접 클릭하여 발송하며,
자동/스케줄러 발송은 구현되어 있지 않습니다.


[활용 시나리오 - 로그인부터 메시지 발송까지]

① 간사가 로그인 페이지(https://ksea-m.vercel.app/login) 접속
   → "카카오로 로그인" 버튼 클릭
   → profile_nickname, friends, talk_message 동의 후 로그인 완료

② 로그인 직후 시스템이 자동으로 동의 항목을 체크하여, 누락 시
   카카오 재동의 화면으로 리다이렉트 (GET /v2/user/scopes 호출)

③ 상단 네비게이션 "카카오 매칭" 메뉴 클릭
   → /kakao-match 페이지 진입
   → "친구 목록 새로고침" 버튼 클릭
   → GET /v1/api/talk/friends 호출하여 본인의 카카오 친구 조회
   → 시스템에 등록된 검토위원 목록과 친구를 1:1 매칭
   → 각 검토위원의 kakao_uuid를 DB에 저장

④ 상단 네비게이션 "알림 현황" 메뉴 클릭
   → /notifications 페이지 진입
   → "새 알림 발송" 버튼 클릭 (다이얼로그 오픈)

⑤ 다이얼로그에서 메시지 템플릿 선택 (검토 요청/도서 접수/리마인더)
   → 제목·본문 확인 및 필요시 수정
   → 수신자 다중 선택 (매칭된 검토위원 목록에서 체크박스)
   → "N명에게 발송" 버튼 클릭

⑥ 버튼 클릭 시 서버에서 POST /v1/api/talk/friends/message/default/send
   호출 (5명씩 배치 분할, pair당 일간 20건 제한 자체 체크)
   → 본인에게 발송하는 경우 POST /v2/api/talk/memo/default/send 사용

⑦ 발송 결과가 화면에 성공·실패 수신자별로 분리 표시
   → NotificationLog 테이블에 발송 이력 저장 → /notifications 현황에서 조회 가능


[특이사항]
- 발송은 100% 간사의 수동 버튼 클릭으로만 이루어집니다.
- 수신자는 사전에 본 시스템 회원가입 + 동의 + 간사와 카카오톡 친구
  관계가 필수이며, 미충족 시 발송 불가합니다.
- pair(간사↔검토위원)당 20건/일 제한을 서버에서 자체 모니터링합니다.
- 메시지 내용은 업무 사실 통지이며 광고성 내용이 없습니다.
```

**글자 수**: 약 1,200자

## 1. 서비스 개요

### 1.1. 서비스명
**건축구조안전 모니터링 통합 시스템**

### 1.2. 서비스 URL
- 운영: https://ksea-m.vercel.app
- 로그인 페이지: https://ksea-m.vercel.app/login

### 1.3. 서비스 성격
공공 안전 관련 업무용 내부 협업 시스템. 일반 대중을 대상으로 하는 서비스가 아니며, 업무 담당자와 전문 검토위원 간의 업무 알림을 목적으로 한다.

### 1.4. 서비스 도입 배경
건축구조 안전 검토는 대한민국 건축법에 따라 **특수구조 건축물, 고층 건축물, 다중이용 건축물** 등 고위험군 건축물에 대해 반드시 수행되어야 하는 법정 절차이다. 본 시스템은 이러한 검토 업무를 수행하는 팀이 수십~수백 건의 검토 대상을 체계적으로 관리하고, 검토위원들과의 업무 커뮤니케이션을 효율화하기 위해 구축되었다.

기존 이메일/전화 기반 업무 전달은 검토위원의 응답률이 낮고 업무 누락이 빈번하여, 검토위원들이 일상적으로 사용하는 **카카오톡을 통한 업무 알림 채널**이 반드시 필요하다.

## 2. 사용자 구성

| 역할 | 인원 | 설명 |
|---|---|---|
| 모니터링 팀장 | 1명 | 전체 업무 총괄 |

| **합계** | **약 60명** | — |

모든 사용자는 **본 시스템에 회원가입된 업무 관계자**이며, 일반 대중이나 불특정 다수에게 발송하지 않는다.

## 3. 카카오 API 사용 목적

### 3.1. 친구 목록 API (`friends` scope)

**필요한 이유:**
- 간사(발신자)는 업무상 사전에 검토위원(수신자)들과 카카오톡 친구 관계이다.
- 친구 목록 API를 통해 시스템 내에 회원가입한 검토위원의 카카오 UUID를 획득하여, 이후 메시지 발송 시 사용한다.
- 친구 UUID는 **본 시스템 내부의 User 테이블에 저장되어 영구 매칭**되며, 매칭 작업은 간사가 수동으로 진행한다 (`/kakao-match` 화면).

**사용하지 않는 용도:**
- 친구 목록을 외부에 노출하거나 판매하지 않음
- 친구 관계 정보를 기반으로 한 추천 기능 없음
- 본인 동의 없이 친구에게 알림을 보내는 행위 없음

### 3.2. 메시지 API (`talk_message` scope)

**필요한 이유:**
검토위원은 본인이 담당하는 건축물의 설계도서 접수, 보완 요청, 검토 마감 등 업무 상태 변화를 **적시에 인지해야 한다**. 이메일은 확인율이 낮고, 문자는 단방향이며 과금이 발생한다. 검토위원들이 일상적으로 사용하는 카카오톡이 가장 효과적인 업무 알림 채널이다.

**발송 시나리오:**
| 시나리오 | 트리거 | 발송 주체 | 내용 |
|---|---|---|---|
| 검토 대상 배정 | 간사가 배정 완료 시 | 간사 (수동 버튼) | "[모니터링] 검토 요청: {건축물명} 검토가 배정되었습니다" |
| 설계도서 접수 안내 | 도서 접수 처리 시 | 간사 (수동 버튼) | "[모니터링] 예비검토서 접수: {건축물} 설계도서가 웹하드에 업로드되었습니다" |
| N차 보완 요청 | 부적합 판정 시 | 간사 (수동 버튼) | "[모니터링] {n}차 보완 요청: {사유}" |
| 검토 마감 임박 | 간사가 리마인더 발송 | 간사 (수동 버튼) | "[모니터링] 검토 마감일이 임박했습니다" |

## 4. 발송 방식 및 트리거

### 4.1. 수동 트리거 원칙
**모든 메시지 발송은 간사가 화면에서 직접 발송 버튼을 클릭하는 방식으로만 동작한다.** 자동 발송, 스케줄러에 의한 일괄 발송은 현재 구현되어 있지 않다.

### 4.2. 발송 화면 흐름
1. 간사 로그인 (카카오 또는 이메일)
2. `/notifications` 페이지 진입
3. "새 알림 발송" 버튼 클릭
4. 수신자 다중 선택 (이미 친구 매칭된 사용자 목록에서)
5. 메시지 템플릿 선택 → 내용 확인
6. **"N명에게 발송" 버튼 클릭**
7. 결과 확인 (성공/실패 수신자 분리 표시)

### 4.3. 자기 자신에게 발송
간사가 본인에게 업무 알림을 남기는 경우 카카오의 **"나에게 보내기"(memo)** API를 사용한다. 친구 관계/UUID 매칭이 불필요하여 별도로 처리한다.

## 5. 사용자 동의 플로우

### 5.1. 동의 항목
사용자가 카카오 로그인 시 다음 항목에 동의한다:
- `profile_nickname` (필수) — 사용자 식별용
- `friends` (필수) — 친구 매칭용
- `talk_message` (필수) — 업무 알림 발송용

### 5.2. 재동의 처리
기존 사용자가 상기 항목에 미동의 상태로 기록된 경우, **로그인 시 자동으로 카카오 재동의 화면으로 리다이렉트**한다. 동의 없이는 친구 매칭 및 알림 발송 기능을 사용할 수 없도록 시스템에서 차단한다.

### 5.3. 동의 항목 진단
간사는 `/kakao-match` 페이지에서 각 사용자의 동의 상태를 조회할 수 있다 (`/v2/user/scopes` API 사용). 미동의 사용자는 **진단 결과가 시스템에 명확히 표시**되어, 해당 사용자에게 재동의를 안내한다.

## 6. 발송 빈도 및 쿼터 준수

### 6.1. 예상 발송 빈도
- **1명의 간사당 일평균 발송 건수**: 약 10~30건 (업무 상황에 따라 변동)
- **전체 일평균 발송 건수**: 약 50~150건 (간사 5명 기준)
- **pair(간사↔검토위원) 당 일간 발송 건수**: 평균 1~2건, 최대 5건

### 6.2. 쿼터 제한 준수 로직
카카오 정책상 pair당 20건/일 제한이 있으며, 시스템에서 **발송 직전에 NotificationLog 테이블의 당일 발송 이력을 조회하여 초과 수신자는 자동 제외**한다 (`backend/routers/notifications.py`의 `PAIR_DAILY_LIMIT` 상수 및 `pair_counts` 로직).

### 6.3. 배치 발송 제한
카카오 권장사항에 따라 **1회 호출당 최대 5명씩 분할 발송**하며, 호출 사이에 적절한 간격을 둔다.

## 7. 개인정보 및 보안

### 7.1. 저장하는 카카오 정보
- `kakao_id` (사용자 식별 및 로그인 유지)
- `kakao_uuid` (친구 UUID, 메시지 발송 대상 식별)
- `access_token`, `refresh_token` (만료 시 자동 갱신)
- `profile_nickname` (사용자 이름 표시)

### 7.2. 저장하지 않는 정보
- 친구의 프로필 썸네일 이미지 URL (일시 표시 후 저장 안 함)
- 카카오 이메일은 선택적으로 저장 (현재 저장 안 함)

### 7.3. 정보 삭제
사용자 탈퇴 시 관련 카카오 정보 일체 삭제 가능.

## 8. 유사 서비스 및 차별점

본 시스템은 **B2B 업무 자동화 도구**이며, 다음 특징을 가진다:
- 가입 사용자가 제한적 (약 60명, 업무 관계자만)
- 모든 발송은 수동 트리거 (자동 스팸 발송 불가)
- 발송 내용이 업무 관련 사실 통지 (광고성 메시지 없음)
- pair당 일간 20건 쿼터 자체적으로 모니터링
- 수신자가 원하지 않으면 간사에게 요청하여 매칭 해제 가능
