# 운영 dry-run 절차

> **목적**: 50명 일괄 온보딩 전에 테스트 계정 2~3개로 전체 흐름을 끝까지 확인.
> **소요 시간**: 약 30분.
> 실패 시 [troubleshooting.md](./troubleshooting.md) 참조.

## 준비

| 역할 | 이메일 | 카카오 친구? | 비고 |
|---|---|---|---|
| 발신 간사 | (본인) | — | 카카오 OAuth 완료 |
| 수신 A | dryrun-a@example.com | ⭕ 친구 등록 | 카카오 발송 흐름 |
| 수신 B | dryrun-b@example.com | ❌ 친구 아님 | manual fallback 흐름 |
| (선택) 수신 C | (실 사용자 1명) | — | 실 운영 검증 |

## 1. 사전 점검

- [ ] 발신 간사로 `/admin` 진입
- [ ] 상단 카카오 동의 배너: 모두 OK 확인
- [ ] **"친구 목록 새로고침"** 클릭 → A의 카카오 닉네임이 친구 목록에 보임 확인

## 2. 단건 등록 + 단건 발송 (A)

- [ ] **"사용자 등록"** 클릭 → A 정보 입력 → 등록
- [ ] 등록 다이얼로그 닫기 (초기 비번은 사용 X — 다음 단계에서 무효화됨)
- [ ] A 행에서 **"매칭"** 클릭 → 카카오 친구 목록에서 A 선택
- [ ] A 행에서 **"초대 발송"** 클릭 → confirm
- [ ] 응답: `delivery=kakao` → alert "카카오 메시지로 초대 링크 발송"
- [ ] **A 카카오톡 확인**: 발신 간사로부터 "비밀번호 설정" 메시지 수신
- [ ] A가 메시지 링크 클릭 → `/setup-password` 접근
- [ ] A가 새 비번 입력 → 완료 화면 → "로그인하러 가기"
- [ ] A가 새 비번으로 로그인 시도 → 성공 + 대시보드 진입

## 3. 단건 등록 + manual 발송 (B)

- [ ] **"사용자 등록"** 클릭 → B 정보 입력 → 등록
- [ ] B 행에서 **"초대 발송"** 클릭 (매칭 없음)
- [ ] 다이얼로그에 setup_url 노출 + 만료시간(72h 후) 표시
- [ ] **"복사"** 버튼 → URL 복사 → SMS/이메일로 B에게 전달
- [ ] B가 링크 클릭 → `/setup-password` → 새 비번 입력 → 완료
- [ ] B가 새 비번으로 로그인 → 성공

## 4. 일괄 등록 + 일괄 발송 (엑셀 — 옵션)

- [ ] 테스트용 엑셀 (A=이름, B=이메일, C=`검토위원`, D=전화 — 3행: dryrun-bulk-1/2/3@example.com)
- [ ] `/admin` → "등록 후 자동 초대 발송" 체크박스 ON
- [ ] **"엑셀 일괄등록"** → 파일 선택
- [ ] 결과 다이얼로그:
  - 등록 3명 / 스킵 0명
  - 일괄 발송 결과: 카카오 0 / 수동 3 / 실패 0 (모두 미매칭)
- [ ] **"수동 전달 필요"** 섹션의 **"전체 복사"** → TSV 형식 확인 (`이름\t링크`)
- [ ] (선택) 실제 사용자 1명에게 전달 → 비번 설정 완료 확인

## 4-2. 토큰 만료 후 재발송 (B)

- [ ] B의 setup token expires_at을 강제로 과거로 변경 (DB 직접 또는 72h 대기)
  - 개발자용 SQL: `UPDATE password_setup_tokens SET expires_at = now() - interval '1 hour' WHERE user_id = (SELECT id FROM users WHERE email = 'dryrun-b@example.com');`
- [ ] B가 같은 setup_url 다시 클릭 → "유효하지 않거나 만료된 링크" 화면
- [ ] `/admin`에서 B 행 **"초대 발송"** → 새 setup_url 다이얼로그 표시
- [ ] 이전 토큰은 자동 무효화됨 (이전 링크로 다시 시도해도 401)
- [ ] B가 새 링크로 비번 설정 → 완료

## 4-3. 기존 사용자 비밀번호 재설정 (A)

- [ ] A 행에서 **"초대 발송"** 클릭 (purpose=initial_setup 기본)
  - 또는 **"PW초기화"** 클릭(과도기 fallback — 일회용 비번)
- [ ] A가 새 링크로 비번 변경 → 완료
- [ ] A가 새 비번으로 로그인 → 성공

> 비번 재설정 시 카카오 매칭은 그대로 유지. 카카오 로그인 흐름도 정상.

## 5. 권한 / 데이터 회귀

- [ ] A로 로그인 → `/admin` 접근 시도 → 403 (REVIEWER 차단)
- [ ] A로 `/buildings` 접근 → 본인 담당 건물만 보이는지
- [ ] (검토위원 50명 등록 후) `/admin`에서 발신 간사로 일부 사용자 매칭 → 다음 발송에서 카카오로 처리되는지

## 6. 정리

- [ ] dry-run용 테스트 계정 삭제 (A, B, dryrun-bulk-1~3)
  - `/admin` 사용자 행 **"삭제"** 버튼
  - 또는 SQL: `DELETE FROM users WHERE email LIKE 'dryrun-%@example.com';`
- [ ] 발급된 토큰도 자동 삭제됨 (FK 없음, 토큰 자체는 만료 cron이 정리)
- [ ] 실제 운영 사용자가 dry-run에 포함됐다면 그대로 둠 (실제 활성 계정)

## 검증 체크포인트

- [ ] 카카오 발송 → 사용자 수신 → 링크 클릭 → 비번 설정 → 로그인 (end-to-end)
- [ ] manual fallback URL → 별도 채널 전달 → 사용자 비번 설정 → 로그인 (end-to-end)
- [ ] 일괄 발송 결과 다이얼로그가 50명 시뮬레이션에서도 사용성 OK인지 (스크롤, 전체 복사)
- [ ] Render 로그에서 발송 이벤트 (`event=bulk_invite_kakao_sent`, `event=password_setup_completed`) 확인

dry-run에서 막히면 [troubleshooting.md](./troubleshooting.md) → 그래도 안 풀리면 코드 수정 후 dry-run 재실행.
